[ 2. Artdink 加密策略演進 ]

從 1996 到 2002 共 6 年的設計變遷，加密策略**逐步加強**：

作品 加密類型 2026 破解狀態
LUNA2 (1996) 4-byte cyclic XOR (key: 9A BC DE 00)，僅 ENC 加密 ENC.ENC ITEMNAME.ENC 等 ~40 個檔 ✅ 完全破解（W08 解出 4400+ 字元）
LD3 (2000) 無加密！36 個 .BIN 全 \\0LD3 magic 明文 ✅ 直接讀（1183 字串）
LD4 (2001) 選擇性加密：40 個明文 + 4 個加密（ITEM_GUARD / NORMAL / WEAPON + MonsterPara.BIN） 🟡 部分破解（用 LD3 plaintext attack 解出前 24 byte XOR keystream，byte 24+ 為 stream cipher state-based 仍未破）
第三之書 (2002) 所有 .dat 用 Huffman 壓縮 + 加密（含 ItemNameStr.dat / MonsterNameStr.dat / SpCardNameStr.dat / 4 ItemParam.dat 等 11 檔） ❌ 未破解（LZSS 8 變體全失敗，需反組譯 Lde3.exe 找 cipher init）

密碼學評論：Artdink 加密設計不是專業密碼學，是「擋業餘玩家修改」程度。LD4 用 zero padding 作為 known plaintext 立刻洩漏 24 byte keystream，是新手錯誤。

[ 3. 系列共通設計 DNA ]

從 LD3 (2000) 起，俠客遊系列確立了核心設計 — LD4 跟第三之書都繼承這套 DNA：

3.1 共通最終魔王 — 安哥摩爾（Angra Mainyu，波斯邪神）

  • LUNA2 (1996)：「魔王安哥爾莫亞」內嵌譯名 / 「魔王阿迦魯瑪」資料表譯名（雙譯名差異反映開發過程翻譯不統一）
  • LD3 (2000)：「安哥摩爾」（× 4 變體）— 從 MonsterPara.BIN 提取確認
  • 第三之書 (2002)：作為「魔王」概念延續

3.2 共通 4 元素精靈（火 / 水 / 風 / 土）

所有 4 作都有 4 元素精靈系統，每作各 4 變體（共 16 隻精靈）。在 LD3 / LD4 / 第三之書都對應「卡片陣營屬性」設計。

3.3 共通 4 陣營（光 / 闇 / 混沌 / 秩序）

  • LD3 (2000)：怪物分類已有「光的狂信者 / 闇的狂信者 / 混沌的狂信者 / 秩序的狂信者」
  • LD4 (2001)：沿用
  • 第三之書 (2002)：成為卡片戰鬥的核心陣營（CoCard.dll 4 陣營屬性）

3.4 4-tier sub-id 系統

LD3 怪物名出現 4 次重複（撒旦 × 4 / 天馬 × 4 / 九尾狐 × 4 等）— 暗示有「弱 / 中 / 強 / 王」4 級難度變體。這設計後來在第三之書也延續。

3.5 隨機世界生成器

LD3 確立了「隨機名字 pool + runtime 組合」設計（mt_*.BIN 共 1183 個 unique 字串）。LD4 沿用並擴充為 1200 字串 + Web/Models HTML 模板。每次新遊戲產生獨一無二的世界內容。

[ 4. 跨檔翻譯不統一現象 ]

系列多處顯示「台灣版直接從日文版改翻譯」的痕跡：

4.1 LUNA2 boss 雙譯名

同 6 隻終局 boss 在 MONSTERS.ENC 解密版 vs LUNA2.EXE 內嵌譯名**不同**：

  • 貝利亞 (ENC) vs 培利亞耳 (EXE)
  • 提斯卡托 (ENC) vs 帖斯卡特利波加 (EXE)
  • 魔王阿迦魯瑪 (ENC) vs 魔王安哥爾莫亞 (EXE)

反映**開發過程翻譯不統一** — 戰鬥畫面、NPC 對話、劇情敘述用不同來源的譯名。

4.2 LD4 Web/Models 模板日文助詞殘留

placeholder 留有日文助詞「の / ライセンス / アイテム / ハンドル」未翻譯（cp950 顯示為「ソ / ヲьиュЧヱ / ヤユЪу / г⑦Эю」）— 譯者沒翻完整。

4.3 MONSTER.DAT byte[8-15] 日文原文殘留

俠客遊 II MONSTER.DAT 含「中文名 + 日文原名（Shift-JIS）」雙欄位，只有 18 boss 級怪物有日文原文（一般人類組全 0）— 證明 boss 命名是「日文先寫 → 翻譯後填入」流程。

[ 7. 已破解 vs 未破解 ]

已完全破解

  • ✅ LUNA2 ENC 加密（4-byte XOR）
  • ✅ LUNA2 MONSTER.DAT (128 records × 104 bytes，13 + 日文原名欄位)
  • ✅ LUNA2 ITEMDATA.MST (456 物品 × 48 bytes，28 個欄位)
  • ✅ LUNA2 MGCDATA.MST (80 records × 32 bytes，49 有效 + 31 空 slot)
  • ✅ LUNA2 HISSATSU.MST (32 records，含 byte[16] = boss 怪物 ID)
  • ✅ LUNA2 PLT 調色盤（GRB444 編碼）
  • ✅ LUNA2 DDT mode 2-39 (4-plane planar EGA)
  • ✅ LD3 全 36 BIN（明文，1183 字串）
  • ✅ LD4 40 BIN（明文，1200 字串）
  • ✅ Lde3.exe 內嵌字串（215 道具 / 75 怪物 / 40 卡片 / 660 NPC 名 等）

部分破解

  • 🟡 LD4 4 加密 BIN 前 24 byte XOR keystream（W56 known plaintext attack）
  • 🟡 lde3s.le3 結構分區（0x00-0x6B header / 0x6C-0x18F 半固定 24 bytes 變動 / 0x190+ MFC CArchive 動態）
  • 🟡 第三之書 .dat Huffman header 結構（HF magic / 256 length table 推測）

仍未破解（需要外部條件）

  • ❌ LD4 byte 24+ 加密 stream cipher state（需反組譯 LD4.exe）
  • ❌ 第三之書 .dat 完整 Huffman 解碼（需反組譯 Lde3.exe 找 dat decoder function）
  • ❌ LD3 / LD4 MSG 加密（XOR 8 keys 全失敗）
  • ❌ TIL sprite 完整解碼為 PNG（圖像格式 + 創作性表現考量）
  • ❌ DD9 完整像素圖（部分背景 fill 已知，但 dim 未知）

[ 8. 研究方法論 ]

有效方法

  1. 跨檔比對：同一 record 結構在不同版本（LD3 vs LD4）對比，找出 patch 差異
  2. 密度分布掃描：用 4KB / 16KB chunk 統計字串密度，找高密度區段
  3. 魔法 byte 識別：file header 前 4-8 bytes 的 magic 看格式類型
  4. Known plaintext attack：已知明文結構（如 magic header / 0 padding）反推加密 keystream
  5. byte-by-byte diff：相似存檔（同 slot 不同時刻）找出變動 byte
  6. 多編碼嘗試：cp950 / Big5 / Big5-HKSCS / Shift-JIS（特別是日文原文殘留）

失效方法（從多輪嘗試學到）

  1. 對 .text section 做 cp950 解碼 — 大量 false positive（random bytes 像中文）
  2. 用 LZSS 標準變體解 LD4 / Lde3 加密 — 算法不對
  3. 對 16-bit DOS EXE 用 r2 自動 xref — DOS reloc 處理有限

[ 9. EXE 反組譯結果 ]

用 radare2 (r2) 對 4 個 EXE 做反組譯統計：

9.1 LD4 加密 cipher init 反組譯嘗試

用 r2 自動 xref 找 ITEM_GUARD / ITEM_NORMAL / ITEM_WEAPON / MonsterPara 的 file open 函式 — 失敗。原因：

  • r2 對 PE32 reloc / IAT 自動處理有限
  • UTF-16LE 字串透過 LoadStringW 動態讀取（用 resource ID 而非 byte address push）
  • 無法直接 push imm32 找 xref

完整破解需要：(a) Ghidra GUI 互動分析 (b) IDA Pro 商業工具 (c) 動態分析 Wine + gdb 跑 LD4 看記憶體 trace。

9.2 各 EXE 反組譯狀態

  • LUNA2.EXE：DOS MZ 16-bit，entry point 識別 + DOS 啟動序列（INT 21h API）確認。函式級 game logic 因 16-bit reloc 困難未深入
  • Lde3.exe：dat decoder 函式 + 256-symbol Huffman tree builder 函式定位完成。完整 dat 解碼需手動移植到 Python 或 r2 emulation
  • LD4.exe：6637 函式中含 cipher init / decrypt function，但需更深入互動分析
