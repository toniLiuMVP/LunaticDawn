[ 問題背景 ]

1999 年 Artdink 釋出的俠客遊：前途道標 (Lunatic Dawn: Passage of the Book) 是 Windows 95 時代的 Win32 GUI 程式 — 內部寫死 96 DPI 假設（Win95 標準）。

到了 Windows 10 / 11 時代，遊戲視窗內字體會重疊在一起，看不清楚。這個問題從 Win10 推出（2015）以來困擾老玩家超過 10 年。

另外 2022 年 Steam 重發版只有日文介面 — 對台灣玩家不友善。

[ 本補丁一次解決兩件事 ]

  1. ✅ 字體不重疊（DPI awareness）— 困擾老玩家 10+ 年的視窗內中文字疊在一起問題
  2. ✅ 字夠大（GetStockObject hook + 倍率可調 50–400%）— 4K 螢幕也看得清楚

兩種玩家路徑（🅰️ Steam 玩家 / 🅱️ 1999 老光碟玩家）都用同一套補丁，流程一樣 — 解壓覆蓋原檔即可。

附加效果：對 Steam 玩家額外變繁體中文（Steam 上架僅日文版）；對老光碟玩家額外保護光碟磁軌（補丁含 NoCD，不用每次插光碟）。

[ 微調倍率 ]

用 Notepad 開 ld95.ini，找 [Window] 段：

[Window]
ScalePercent=200

把 200 改成你要的倍率，存檔，重啟 Ld95.exe 即生效（不用重啟系統 / 不用登出）：

ScalePercent倍率適用場景
1001.0x原始大小（跟不裝補丁一樣）
1501.5x1080p 螢幕 / 字稍大

[ 怎麼解決的 — 7 個版本演進故事 ]

這個補丁從 v1 到 v7 經過 7 個版本才攻克，每個失敗都教了我們一件事：

版本嘗試什麼結果學到
v1呼叫 SetProcessDpiAwarenessContext(UNAWARE_GDISCALED)，讓 Win10+ DWM bitmap stretch 接管縮放🟡 字不重疊但字小DWM stretch 需配合 Windows 顯示縮放設定 > 100% 才會放大
v2+ IAT hook user32!CreateWindowExA 把主視窗 nWidth/nHeight × 2🔴 視窗變大但 client area 沒變大 — 留下大片空白邊框MFC 視窗 size 來自 SetWindowPlacement 而非 CreateWindowEx
v3+ IAT hook SetWindowPlacement 攔截 MFC restore🔴 視窗 + client 都大了，但字仍是 96 DPI 小字視窗大小不等於字體大小，GDI 渲染按邏輯像素獨立
v4+ IAT hook gdi32!CreateFontIndirectA 把 lfHeight × scale🔴 字仍沒變大lfHeight=0 時被跳過 — MFC 大宗用 lfHeight=0 讓 GDI 自選
v5修 lfHeight=0 → 強制 -11 再乘 scale🟡 MFC UI 字（功能表 / 按鈕）變大了，但遊戲主畫面字仍沒大遊戲字根本不走 CreateFontIndirectA
v6用 r2 反組譯 Ld95.exe，發現遊戲字真正 path：GetStockObject(SYSTEM_FONT) + SelectObject + TextOutA。Hook GetStockObject 回傳自建大字體🔴 字 + 視窗都消失 — 自建 LOGFONT 用 DEFAULT_CHARSET，GDI 在繁中環境 pick 非中文字體cp950 環境下自建 LOGFONT 必須明確設 charset，否則 Big5 字無法渲染
v7改用 GetObjectA 取原 SYSTEM_FONT 的 LOGFONTA（含正確 charset / faceName），只改 lfHeight 乘 scale，再 CreateFontIndirectA✅ 成功！字大 + 不重疊 + 視窗正常 + UI 字也保持原大小不要自建，從系統原 stock font 拷貝屬性只改要改的

轉捩點是 v6 用 r2 反組譯找出真實字體渲染 path — 此前 v1–v5 都在猜（user32 視窗 hook / CreateFont hook），猜了 5 輪沒中。反組譯 ground truth 給出來後，v6 → v7 兩個版本一發中。

最終解決方案：1 個 IAT hook + 1 個 GetObjectA 呼叫 + 改 1 個欄位。整個 LDWMM.dll 不到 10 KB，純 Win32 API，零 CRT 依賴。

[ 為什麼之前那麼難解 ]

  • Win95 GUI 程式 hardcoded 96 DPI：1999 沒人預期 4K 螢幕，所有像素都寫死
  • MFC 字體渲染分兩條 path：UI 字（功能表 / 按鈕）走 CreateFontIndirectA，遊戲主畫面字走 GetStockObject(SYSTEM_FONT) — 攔錯一條就只放大一半
  • Windows DLL search order 有 KnownDLLs 機制：winmm.dll 永遠從 system32 載入，application directory 的同名 DLL 會被忽略 — 必須改 PE import name 成 LDWMM.dll（9 byte 同長度替換）才能 hook 載入
  • DllMain 階段呼叫複雜 GDI API 可能 fail：v6 失敗根因，OS loader 階段 GDI subsystem 尚未 fully init（v7 學到要從 stock object 拷貝屬性）
  • cp950 / Big5 字符集 trap：自建 LOGFONT 用 DEFAULT_CHARSET，GDI 可能 pick 非中文字體 → 中文字渲染失敗看起來「字消失」

[ 技術原理摘要 ]

  1. 不重疊：DllMain 呼叫 SetProcessDpiAwarenessContext(UNAWARE_GDISCALED)，Win10 1809+ DWM bitmap stretch 處理 DPI scaling
  2. 字夠大：DllMain 內
    a. GetStockObject(SYSTEM_FONT) 取原系統字體 HFONT
    b. GetObjectA 拿出 LOGFONTA（含正確 cp950 charset + 系統內建中文字體 faceName）
    c. 改 lfHeight × ScalePercent / 100
    d. CreateFontIndirectA 建大字體 HFONT 存進全域變數
  3. 運行時攔截：hook gdi32!GetStockObject，當 Ld95 請求 SYSTEM_FONT / DEFAULT_GUI_FONT / OEM_FIXED_FONT 等字體 stock object → 回傳我們建的大字體；非字體 stock object（palette / brush / pen / bitmap）pass-through 不攔
  4. WINMM rename：Ld95.exe PE import name WINMM.dll → LDWMM.dll（9 byte 同長度 byte-level 替換）繞 Windows KnownDLLs，讓 application directory 的 LDWMM.dll 被優先載入
  5. 純 Win32 API 零 CRT 依賴：LDWMM.dll 用 -nostdlib -nodefaultlibs 編譯，避免跟 MFC 內建 runtime 衝突
